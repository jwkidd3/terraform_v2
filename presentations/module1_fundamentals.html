‚öôÔ∏è Terraform Architecture: Under the Hood

Core Components Explained

Terraform Core
  • Terraform Binary: The main executable that orchestrates everything
  • Configuration Files: .tf files containing your infrastructure definitions
  • State File: The source of truth for your infrastructure
  • Providers: Plugins that interact with APIs (AWS, Azure, GCP, etc.)
  • Resources: Individual infrastructure components
  • Data Sources: Read-only information from existing infrastructure

Terraform Execution Flow

1. Configuration Parsing 2. Provider Initialization 3. Resource Graph Creation 4. State Comparison 5. Plan Generation 6. Resource Provisioning 7. State Update

üìù HCL Syntax: Basic Structures

Basic Block Structure

resource "aws_instance" "example" {
  ami           = "ami-0e820afa569e84cc1"  # us-east-2 example - use dynamic lookups in production
  instance_type = "t2.micro"
  
  tags = {
    Name        = "ExampleInstance"
    Environment = "Production"
    Owner       = "DevOps Team"
  }
}

Key HCL Components

  • Blocks: Resources, Data, Variables, Outputs
  • Arguments: Name-value pairs within blocks
  • Expressions: Values that produce results
  • Comments: # single line, /* multi-line */

Variable Types

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t2.micro"
}

variable "tags" {
  description = "Resource tags"
  type        = map(string)
  default     = {}
}

variable "availability_zones" {
  description = "List of AZs"
  type        = list(string)
}

Basic Data Types

  • string: Text values
  • number: Numeric values
  • bool: true or false
  • list: Ordered collection
  • map: Key-value pairs

  type = string

üîÑ Terraform Workflow: From Code to Infrastructure

1. Initialization Phase

$ terraform init

What happens:

  • Downloads required providers
  • Initializes backend configuration
  • Prepares working directory
  • Validates configuration syntax

Advanced Init Options

# Upgrade providers $ terraform init -upgrade # Copy from another environment $ terraform init -from-module=../prod # Backend configuration $ terraform init -backend-config="bucket=my-state-bucket"

üí° Best Practice: Always run init when joining a project or after adding new providers.

2. Planning Phase

$ terraform plan

Plan Output Analysis:

  • + create - New resources
  • ~ modify - Changed resources
  • - destroy - Removed resources
  • ‚üµ read - Data sources

Plan Variations

# Save plan for later $ terraform plan -out=tfplan # Target specific resources $ terraform plan -target=aws_instance.web # Destroy plan $ terraform plan -destroy # Variable specification $ terraform plan -var="instance_count=3"

‚ö†Ô∏è Always review plans carefully! Plans show exactly what changes will be made.

3. Apply & Destroy

$ terraform apply $ terraform apply tfplan $ terraform apply -auto-approve

Apply Process:

  • Shows plan for confirmation
  • Creates/modifies/destroys resources
  • Updates state file
  • Shows summary of changes

Destroy Operations

# Destroy everything $ terraform destroy # Destroy specific resource $ terraform destroy -target=aws_instance.test # Plan destroy first $ terraform plan -destroy $ terraform apply -destroy

State Management

# View state $ terraform state list $ terraform state show aws_instance.web # Import existing resources $ terraform import aws_instance.web i-1234567890abcdef0 # Move resources $ terraform state mv aws_instance.web aws_instance.web_server

üîó Resource Management: Dependencies and Relationships

Implicit Dependencies

# VPC creates first, then subnet
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

resource "aws_subnet" "web" {
  vpc_id     = aws_vpc.main.id  # ‚Üê Implicit dependency
  cidr_block = "10.0.1.0/24"
}

Explicit Dependencies

resource "aws_instance" "web" {
  ami           = "ami-0e820afa569e84cc1"  # us-east-2 example - use dynamic lookups in production
  instance_type = "t2.micro"
  
  # Force dependency even without reference
  depends_on = [
    aws_security_group.web,
    aws_key_pair.deployer
  ]
}

Resource Addressing

Resource Address Format:

resource_type.resource_name[index]
  • aws_instance.web - Single resource
  • aws_instance.web[0] - First in count
  • aws_instance.web["primary"] - for_each key

Resource Meta-Arguments

resource "aws_instance" "web" {
  count = 3  # Create 3 instances
  
  ami           = "ami-0e820afa569e84cc1"  # us-east-2 example - use dynamic lookups in production
  instance_type = "t2.micro"
  
  # Lifecycle management
  lifecycle {
    create_before_destroy = true
    prevent_destroy       = true
    ignore_changes       = [ami, user_data]
  }
  
  tags = {
    Name = "web-${count.index + 1}"
  }
}

Advanced Resource Patterns

# for_each for dynamic resources
variable "users" {
  type = map(object({
    role = string
    team = string
  }))
  
  default = {
    "alice" = { role = "admin", team = "platform" }
    "bob"   = { role = "user", team = "dev" }
  }
}

resource "aws_iam_user" "users" {
  for_each = var.users
  
  name = each.key
  
  tags = {
    Role = each.value.role
    Team = each.value.team
  }
}

Conditional Resource Creation

# Create resource only in production
resource "aws_cloudtrail" "audit" {
  count = var.environment == "production" ? 1 : 0
  
  name           = "${var.project_name}-audit-trail"
  s3_bucket_name = aws_s3_bucket.audit[0].bucket
}

Naming Conventions

Resource Naming Standards:

  • Resources: resource "aws_instance" "web_server"
  • Variables: var.vpc_cidr_block
  • Outputs: output "instance_public_ip"
  • Locals: local.common_tags
  • Tags: Name = "${var.project}-${var.environment}-web"

Security Best Practices

üîí Security Essentials:

  • Never hardcode secrets in configuration
  • Use environment variables for sensitive data
  • Implement least privilege access
  • Encrypt state files
  • Use remote backends with locking
  • Regular security scanning of configurations

Enterprise Patterns

üè¢ Enterprise Success Patterns:

  • Module Strategy: Reusable, tested, versioned modules
  • Environment Separation: Isolated state and configurations
  • GitOps Workflow: Code review, automated testing, deployment
  • Policy as Code: Automated governance and compliance
  • Monitoring Integration: Track infrastructure changes

Common Pitfalls to Avoid

‚ùå Common Mistakes:

  • State File Conflicts: Always use remote state for teams
  • Resource Drift: Regular plan checks to detect changes
  • Hardcoded Values: Use variables and data sources
  • No Backup Strategy: State files must be backed up
  • Ignoring Drift: Address configuration differences immediately
  • Poor Module Design: Modules should be focused and reusable

  description = "Environment name"

# main.tf
provider "aws" {
  region = var.aws_region
  
  default_tags {
    tags = local.common_tags
  }

# VPC and Networking
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true
  
  tags = {
    Name = "${local.name_prefix}-vpc"
  }
}

resource "aws_subnet" "public" {
  count = length(data.aws_availability_zones.available.names)
  
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.${count.index + 1}.0/24"
  availability_zone       = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true
  
  tags = {
    Name = "${local.name_prefix}-public-${count.index + 1}"
    Type = "Public"
  }
}

# Security Group
